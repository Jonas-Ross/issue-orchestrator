# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Tauri 2 + Preact desktop app that runs N Claude Code `issue-team` sessions in
one window, each inside its own `git worktree`. Per-tab status pills are
driven by Claude Code's hook system over a Unix domain socket. macOS-only.

## Common commands

```bash
cargo tauri dev                                                  # canonical dev loop (NOT npm run dev alone)
cargo test --manifest-path src-tauri/Cargo.toml --lib            # all backend tests
cargo test --manifest-path src-tauri/Cargo.toml --lib spawn::    # one module
npm run bindings                                                 # regenerate src/lib/bindings.ts (also runs via predev/prebuild)
npm run build                                                    # tsc + vite production build
```

`cargo tauri dev` invokes `npm run dev` via `beforeDevCommand`, which itself
runs `predev` â†’ `npm run bindings`, so the TS bindings stay in sync with the
Rust IPC surface on every launch.

## Architecture: the parts you can't grasp from a single file

### Actor-owned session registry (`src-tauri/src/registry/`)

`SessionRegistryActor` owns the `HashMap<SessionId, Session>` outright. All
state mutations go through `RegistryCmd` messages on a `tokio::mpsc` mailbox
(`Spawn`, `Write`, `Resize`, `Kill`, `List`, `HookEvent`). There is no shared
`Mutex<HashMap>` â€” locking and ordering bugs are designed away. The actor
publishes `RegistryEvent`s on an unbounded mpsc; it never touches `AppHandle`.

### Domain â†’ Tauri event bridge (`src-tauri/src/ipc/mod.rs`)

`ipc::spawn_event_bridge` is the only thing that knows how to translate a
`RegistryEvent` into a typed Tauri event (`PtyData`, `SessionAdded`,
`SessionRemoved`, `StatusChange`). This is what lets the registry, hook
receiver, and spawn flow be unit-tested without a Tauri runtime â€” tests
subscribe to `RegistryEvent` directly.

### Typed IPC contract (`make_specta_builder` in `src-tauri/src/lib.rs`)

Single source of truth for the Rust â†” TS surface. Both runtime startup and
the `bin/export-bindings` binary call it, so `src/lib/bindings.ts`
(gitignored, regenerated by `npm run bindings`) cannot drift from the
`#[tauri::command]` and `#[derive(specta::Type)]` definitions. **When adding
a command or event type, you must also add it to `collect_commands!` /
`collect_events!` in `make_specta_builder`** â€” otherwise the frontend won't
see it.

### Hook bridge (Claude Code â†’ app)

Path: Claude session fires hook â†’ `~/.claude/settings.json` invokes
`<app-data>/hook.sh` (generated from `src-tauri/src/hooks/hook.sh.template`,
rewritten on every app start) â†’ script pipes JSON through `jq -c` to inject
`session_orch_id` from `$ISSUE_ORCH_SESSION_ID` and `nc -U`s it to
`hooks.sock` â†’ `hooks::run_listener` reads the connection (tolerates
multi-line pretty-printed JSON via `serde_json::Deserializer::from_slice
.into_iter`) â†’ `RegistryCmd::HookEvent` â†’ actor maps
`SessionStart/Notification/Stop/SessionEnd` to `Status` and emits
`StatusChange`. Hooks for sessions we didn't spawn (no `session_orch_id`)
are silently dropped.

Correlation key is **`ISSUE_ORCH_SESSION_ID`** (env var injected by
`apply_common_env` in `registry/mod.rs` at PTY spawn time), not `cwd` â€” so
the user can `cd` freely without breaking status. Without `jq` installed,
the listener still parses payloads but correlation is lost.

### Boundary traits for shellouts (`src-tauri/src/spawn.rs`)

`IssueClient` (wraps `gh`) and `GitRunner` (wraps `git`) are traits stored
in `AppState` as `Arc<dyn â€¦>`. Production: `GhCli` / `GitCli`. Tests in
`src-tauri/src/spawn/tests.rs` substitute recording mocks, so spawn-flow
tests touch no real `gh` and create no real worktrees. Same pattern should
be used whenever a new external CLI is shelled out to.

### PTY layer (`src-tauri/src/pty.rs`)

Stateless `spawn_pty` returning `PtyHandles`. The reader runs on
`std::thread::spawn` (portable-pty's `read` is blocking â€” do not move it to
a tokio task). A leftover-byte buffer holds UTF-8 sequences split across
read boundaries so the frontend never sees a truncated multibyte glyph.
`Drop` on `PtyHandles` kills the child, which is what makes Cmd-Q clean up
all sessions.

### Frontend PTY demuxer (`src/state/pty-stream.ts`)

A single module-level listener for `events.ptyData` is started once at boot.
It buffers chunks per `sessionId` until a `<TerminalView>` calls
`attachTerminal`, then drains. This solves the race where the PTY emits
bytes before the React tree has mounted the matching xterm. xterm
`Terminal` instances are heavy: keep them mounted (off-screen for inactive
tabs) rather than recreating on tab switch, and call `term.dispose()` on
session kill â€” not just on unmount.

## Test strategy

Backend has unit tests for the three pillars:
- `registry/tests.rs` â€” actor command round-trips, event emission, real-PTY data flow.
- `hooks/tests.rs` â€” status mapping, JSONL persistence, unknown-session drop.
- `spawn/tests.rs` â€” mocks `IssueClient` and `GitRunner` to cover the
  new-branch, existing-branch, and existing-worktree paths.

There are no frontend tests yet; UI verification is manual via the README's
"Verify M1" / "Verify M3 + M4" checklists.

## Things that are deliberately out of scope

(per `MILESTONES.md`) â€” do not introduce them speculatively: codesigning /
notarization, Windows or Linux support, cross-machine orchestration,
embedded `gh pr` views (link out via `tauri-plugin-shell`'s
`shell:allow-open` capability instead, scoped to `https://github.com/**`),
session persistence across app restart.
